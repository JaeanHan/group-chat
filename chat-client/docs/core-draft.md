# 오픈 채팅방 클라이언트 Core

## 1. 목적

이 문서는 오픈 채팅방 비즈니스 규칙을 사용자 경험과 클라이언트 상태 관점에서 해석하는 기준을 정의한다.

클라이언트 Core는 사용자가 어떤 상태를 인지해야 하는지, 어떤 상태를 조작할 수 있는지, 서버 확정 전 임시 상태를 어디까지 표현할 수 있는지를 정의한다.

## 2. 범위

이 문서는 오픈 채팅방 클라이언트에서 다루는 핵심 사용자 경험과 상태 해석 범위를 정의한다.

포함 범위:

```text
- 오픈 채팅방 목록 조회
- 오픈 채팅방 상세 조회
- 오픈 채팅방 참여
- 채팅방 화면 진입과 이탈
- 메시지 조회
- 메시지 전송
- 메시지 수정
- 메시지 삭제
- 읽음 상태 표시
- unread count 표시
- read receipt count 표시
- WebSocket 연결과 구독
- 재연결과 누락 데이터 복구
- 전송 실패와 동기화 실패 표시
```

제외 범위:

```text
- 관리자 운영 화면
- 파일/이미지 업로드 UX
- 알림/푸시 정책
- 검색
- 신고/제재
- 테마, 디자인 시스템, 스타일 가이드
```

## 3. 핵심 관점

### 3.1 사용자 인지 정합성

사용자가 보는 화면 상태는 서버에 확정된 상태와 일시적으로 다를 수 있다.

다만 사용자는 다음 상태 차이를 명확히 구분할 수 있어야 한다.

```text
- 전송 중
- 전송됨
- 전송 실패
- 재연결 중
- 동기화 중
- 삭제된 메시지
- 접근할 수 없는 room
```

클라이언트는 지연, 재시도, 재연결이 발생하더라도 사용자가 현재 상태를 납득할 수 있도록 표시해야 한다.

### 3.2 상태 소유권

클라이언트는 모든 상태의 source of truth가 아니다.

클라이언트 상태는 다음처럼 구분한다.

```text
server state:
  서버가 확정한 상태

client runtime state:
  현재 화면, 연결, 입력, 스크롤처럼 클라이언트 런타임에서만 의미 있는 상태

optimistic state:
  서버 확정 전 사용자 행동을 즉시 반영하기 위한 임시 상태

derived state:
  server state와 runtime state를 조합해 계산한 화면 상태
```

상태마다 소유권과 복구 기준이 달라야 한다.

### 3.3 동기화

채팅 화면은 query 결과, WebSocket event, optimistic state가 같은 화면 데이터를 함께 변경한다.

클라이언트는 다음 상황을 정상 상황으로 간주하고 처리해야 한다.

```text
- 같은 메시지를 query와 WebSocket event로 모두 받음
- WebSocket event가 늦게 도착함
- WebSocket event가 중복 도착함
- WebSocket 연결이 끊겼다가 재연결됨
- optimistic message와 서버 확정 message를 합쳐야 함
```

최종 화면 상태는 서버에서 다시 조회 가능한 상태와 맞출 수 있어야 한다.

### 3.4 회복 가능성

클라이언트는 실시간 이벤트를 빠른 반영 수단으로 사용한다.

실시간 이벤트가 유실되어도 query 또는 catch-up으로 복구할 수 있어야 한다.

읽음 표시, unread count, read receipt count는 일시적으로 stale할 수 있지만, 서버 기준 조회 결과로 다시 보정할 수 있어야 한다.

### 3.5 지연 허용성

클라이언트는 네트워크 지연과 서버 처리 지연을 사용자에게 숨기기만 하지 않는다.

사용자 행동의 결과가 확정되지 않은 경우, 화면은 확정 상태와 임시 상태를 구분해서 표시해야 한다.

## 4. 채팅방

채팅방은 사용자가 발견하고, 상세를 확인하고, 참여 여부를 결정하는 대상이다.

클라이언트는 채팅방을 다음 사용자 상태로 해석한다.

```text
open:
  사용자가 목록 또는 상세에서 확인할 수 있고 참여할 수 있는 room

joined:
  사용자가 참여 중이며 메시지 화면에 접근할 수 있는 room

left:
  사용자가 나갔고 더 이상 메시지 화면에 접근하지 않는 room

unavailable:
  종료되었거나 접근할 수 없는 room

managed:
  운영 개입 중일 수 있으나 기존 참여자의 사용은 제한되지 않을 수 있는 room
```

채팅방 목록과 상세 화면은 서버에서 조회 가능한 room만 보여준다.

이미 화면에 있던 room이 더 이상 접근 불가 상태가 되면, 클라이언트는 stale 화면을 유지하지 않고 접근 불가 상태로 전환해야 한다.

## 5. 참여자

참여자는 사용자가 특정 채팅방에서 무엇을 할 수 있는지를 결정하는 기준이다.

클라이언트는 참여 상태를 다음 화면 의미로 해석한다.

```text
not joined:
  room 상세를 볼 수 있지만 메시지 화면에는 접근하지 않은 상태

joined:
  메시지를 조회하고 보낼 수 있는 상태

left:
  room을 나갔고 메시지 화면 접근이 종료된 상태
```

사용자가 room을 나가면 클라이언트는 즉시 해당 room의 실시간 구독과 읽음 세션을 정리해야 한다.

재입장한 사용자는 이전 참여 구간의 메시지를 볼 수 없을 수 있으므로, 클라이언트는 과거 캐시를 현재 메시지 목록에 섞지 않아야 한다.

## 6. 메시지

메시지는 채팅방 화면에서 사용자가 가장 직접적으로 인지하는 상태다.

메시지는 사용자에게 다음 상태로 보일 수 있다.

```text
sending:
  사용자가 전송했지만 서버 확정 전인 메시지

sent:
  서버에 저장이 확정된 메시지

failed:
  전송 실패로 사용자가 재시도 또는 삭제할 수 있는 메시지

edited:
  수정된 메시지

deleted:
  content를 노출하지 않는 삭제 표시 메시지
```

메시지의 최종 순서는 서버가 부여한 room 단위 순서를 기준으로 한다.

서버 확정 전 메시지는 임시 순서로 표시할 수 있지만, 서버 확정 후에는 서버 순서에 맞춰 재배치될 수 있다.

같은 메시지가 query와 WebSocket event로 모두 도착해도 클라이언트는 중복 렌더링하지 않아야 한다.

삭제된 메시지는 일반 메시지 content처럼 표시하지 않는다.

## 7. 읽음 상태와 표시

읽음 상태는 사용자가 메시지를 어디까지 확인했는지, 다른 참여자가 내 메시지를 어느 정도 확인했는지를 표현한다.

클라이언트는 읽음 상태를 다음 사용자 의미로 해석한다.

```text
unread count:
  사용자가 아직 읽지 않은 메시지 수

read receipt count:
  특정 메시지를 읽은 참여자 수

active read session:
  사용자가 현재 room 화면을 보고 있어 읽음 처리 대상이 될 수 있는 상태
```

읽음 표시는 실시간으로 갱신될 수 있지만, 실시간 이벤트만으로 최종 진실을 판단하지 않는다.

읽음 이벤트가 유실되거나 재연결이 발생하면 서버 조회 결과로 보정한다.

사용자가 같은 room을 여러 connection에서 보고 있어도 read receipt 표시는 한 명의 사용자 기준으로 해석되어야 한다.

## 8. 실시간 연결

실시간 연결은 채팅방 화면을 빠르게 갱신하기 위한 수단이다.

클라이언트는 연결 상태를 다음 사용자 상태로 표현할 수 있어야 한다.

```text
connecting:
  실시간 연결을 준비하는 상태

connected:
  실시간 이벤트를 받을 수 있는 상태

reconnecting:
  실시간 연결이 끊겨 복구 중인 상태

syncing:
  누락된 데이터를 조회해 화면을 보정하는 상태

offline:
  실시간 갱신을 받을 수 없는 상태
```

WebSocket 연결이 끊겨도 클라이언트는 즉시 모든 화면 상태를 폐기하지 않는다.

연결이 끊긴 동안 사용자는 stale한 데이터를 볼 수 있으며, 클라이언트는 해당 상태를 명확히 표시해야 한다.

재연결 후 클라이언트는 마지막으로 처리한 메시지 순서 이후의 데이터를 조회해 화면을 보정한다.

## 9. 실패 표시

실패는 사용자 행동 실패와 시스템 동기화 실패로 나누어 표현한다.

```text
user action failure:
  메시지 전송 실패, 수정 실패, 삭제 실패처럼 사용자가 수행한 동작이 실패한 상태

sync failure:
  재연결 실패, catch-up 실패, unread count 갱신 실패처럼 화면 동기화가 실패한 상태
```

사용자 행동 실패는 사용자가 재시도하거나 취소할 수 있어야 한다.

동기화 실패는 화면 전체를 막기보다, 가능한 범위에서 stale 상태를 표시하고 복구를 시도한다.

## 10. 클라이언트가 보장할 것

클라이언트는 다음을 보장해야 한다.

```text
- 사용자가 현재 room에서 무엇을 할 수 있는지 명확히 보여준다.
- 메시지 전송 중, 성공, 실패 상태를 구분한다.
- 같은 메시지를 중복 렌더링하지 않는다.
- 메시지는 room 안에서 순서가 뒤집혀 보이지 않도록 정렬한다.
- 연결 끊김과 재연결 중 상태를 표시한다.
- 재연결 후 누락된 메시지를 복구한다.
- 삭제된 메시지의 content를 일반 메시지처럼 노출하지 않는다.
- 읽음 표시는 유실 가능성을 고려해 서버 조회 결과로 보정할 수 있어야 한다.
```

## 11. 클라이언트가 보장하지 않을 것

클라이언트는 다음을 최종적으로 보장하지 않는다.

```text
- 사용자의 최종 참여 권한
- 메시지 수정 가능 여부의 최종 판단
- 메시지 삭제 가능 여부의 최종 판단
- unread count의 최종 진실
- read receipt count의 최종 진실
- WebSocket event의 exactly-once 수신
- 모든 사용자가 같은 시점에 같은 화면을 보는 것
```

이 값들은 서버 확정 상태 또는 서버 조회 결과를 기준으로 보정한다.

```text

  오픈 채팅방 개념을 클라이언트 관점의 사용자 의미로 해석한다.

state-model.md:
  server state, runtime state, optimistic state, derived state를 구분한다.

user-flows.md:
  사용자 행동별 화면 변화와 상태 변화를 정의한다.

sync.md:
  query, WebSocket event, optimistic state를 병합하고 복구하는 전략을 정의한다.

ui-states.md:
  사용자에게 노출되는 구체적인 UI 상태를 정의한다.
```
