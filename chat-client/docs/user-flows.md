# 오픈 채팅방 클라이언트 User Flows

## 1. 목적

이 문서는 오픈 채팅방 클라이언트에서 사용자가 어떤 페이지 흐름을 통해 room을 발견하고, 참여하고, 메시지를 주고받는지 정의한다.

---

## 2. 페이지 범위

오픈 채팅방 클라이언트는 다음 3개 페이지를 기준으로 한다.

```text
1. 내가 참여한 방 목록 페이지
2. 참여 가능한 방 조회 페이지
3. 방 채팅 페이지
```

방 상세는 별도 페이지로 두지 않는다.

방 상세는 참여 가능한 방 조회 페이지에서 summary, modal, popup 등의 형태로 표현할 수 있다.

로그인, 회원가입, 계정 관리 페이지는 이 문서의 범위에 포함하지 않는다.

---

## 3. 공통 흐름 원칙

사용자는 서버 도메인 상태를 직접 다루지 않는다.

클라이언트는 외부 상태와 사용자 맥락을 조합해 현재 페이지에서 가능한 행동을 보여준다.

사용자 행동은 다음 흐름으로 처리한다.

```text
사용자 행동
  -> 현재 페이지 맥락 확인
  -> 필요한 정보 확인
  -> 행동 가능 여부 판단
  -> 필요한 조회 또는 요청 실행
  -> 진행 중 상태 표시
  -> 결과에 따라 다음 흐름으로 이동
```

사용자 행동이 실패하면 기존 화면을 즉시 폐기하지 않는다.

실패한 행동은 사용자가 재시도하거나 다른 행동을 선택할 수 있는 상태로 표현한다.

실시간 이벤트는 빠른 반영 수단이다.

재조회와 누락 복구 결과는 실시간 이벤트 유실, 중복, 지연으로 생긴 화면 차이를 보정한다.

이 문서에서 읽음 관련 표시는 다음 용어로 구분한다.

```text
room unread count:
  현재 사용자가 room에서 아직 읽지 않은 메시지 수.
  내가 참여한 방 목록 페이지의 badge에 사용한다.

message read receipt count:
  특정 room의 하나의 메시지를 현재 ACTIVE member 중 몇 명이 읽었는지 나타내는 수.
  방 채팅 페이지의 메시지 옆 표시에 사용한다.
  안 읽은 사람 수가 필요하면 클라이언트가 message read receipt count와 ACTIVE member snapshot을 기준으로 파생한다.
```

## 4. 내가 참여한 방 목록 페이지

내가 참여한 방 목록 페이지는 사용자가 이미 참여 중인 room으로 빠르게 돌아가기 위한 페이지다.

이 페이지의 핵심 가치는 사용자가 현재 대화 흐름을 빠르게 파악하고, room unread count가 있는 room으로 진입할 수 있게 하는 것이다.

### 4.1 진입 흐름

```text
사용자가 내가 참여한 방 목록 페이지에 진입
  -> 참여 중인 room 목록 조회
  -> room unread count 보정
  -> room summary 표시
```

목록 조회에 실패해도 기존에 표시 중이던 목록을 즉시 비우지 않는다.

서버가 room 종료, 접근 불가, 참여 상태 상실을 명확히 알려준 경우에만 해당 room의 사용 가능 상태를 보정한다.

### 4.2 표시 정보

내가 참여한 방 목록에서는 사용자가 이미 참여 중인 room의 현재 대화 상태를 보여준다.

```text
표시 대상:
  room name
  participantCount
  마지막 메시지 미리보기
  room unread count
  운영 상태 표시가 필요한 경우

표시하지 않는 대상:
  참여 전 room의 메시지 내용
  참여 전 room의 room unread count
```

마지막 메시지 미리보기와 room unread count는 내가 참여한 room의 목록에서만 의미를 가진다.

참여하지 않은 room의 메시지 preview나 room unread count를 이 페이지에서 표현하지 않는다.

### 4.3 주요 행동

```text
방 선택:
  해당 room의 채팅 페이지로 진입한다.

목록 새로고침:
  참여 중인 room 요약 정보와 room unread count를 다시 확인한다.

방 나가기:
  방 나가기를 요청한다.
  성공하면 해당 room을 참여 중인 목록에서 제거하거나 나간 상태로 보정한다.
```

방 채팅 페이지로 진입하기 전, 클라이언트는 현재 사용자가 해당 room의 채팅 화면을 사용할 수 있는지 확인해야 한다.

목록에 표시된 room이라고 해서 항상 채팅 페이지 진입이 가능한 것은 아니다.

### 4.4 room unread count 갱신 흐름

```text
목록 페이지 진입:
  클라이언트가 현재 사용자의 room unread count를 다시 조회한다.

사용자가 목록을 수동 새로고침:
  클라이언트가 room unread count를 다시 조회한다.

앱 foreground 복귀:
  클라이언트가 room unread count를 다시 조회할 수 있다.

polling 시점:
  클라이언트가 room unread count를 주기적으로 다시 조회할 수 있다.

방 채팅 페이지에서 목록으로 복귀:
  클라이언트가 room unread count를 다시 조회할 수 있다.

room unread count 조회 성공:
  클라이언트는 같은 roomId의 room unread count를 서버 응답값으로 덮어쓴다.

room unread count 조회 실패:
  클라이언트는 기존 room unread count를 유지하고 다음 갱신 시점에 다시 시도한다.
```

room unread count는 현재 사용자 기준으로 서버가 계산한 값이다.

클라이언트는 room unread count를 직접 계산하지 않는다.

### 4.5 실패와 보정

```text
목록 조회 실패:
  기존 목록을 유지하고 재시도 가능 상태를 제공한다.

방 진입 실패:
  해당 room의 상세 상태를 다시 확인하도록 유도한다.

방 나가기 실패:
  기존 참여 상태를 유지하고 사용자가 재시도할 수 있게 한다.
```

## 5. 참여 가능한 방 조회 페이지

참여 가능한 방 조회 페이지는 사용자가 아직 참여하지 않은 오픈 채팅방을 발견하고 참여 여부를 결정하는 페이지다.

이 페이지의 핵심 가치는 사용자가 대화 내용이 아니라 room의 맥락을 보고 참여 의사를 결정할 수 있게 하는 것이다.

### 5.1 진입 흐름

```text
사용자가 참여 가능한 방 조회 페이지에 진입
  -> 참여 가능한 room 목록 조회
  -> room summary 표시
  -> 사용자가 room 선택
  -> room 상세를 summary, modal, popup 등으로 표시
  -> 참여 여부 결정
```

방 상세는 별도 페이지가 아니다.

참여 전 사용자는 메시지 목록을 조회하지 않는다.

### 5.2 표시 정보

참여 가능한 방 조회 페이지에서는 room에 참여할지 판단하는 데 필요한 정보만 보여준다.

```text
표시 대상:
  room name
  participantCount
  owner 또는 소개 정보가 필요한 경우
  운영 상태 표시가 필요한 경우

표시하지 않는 대상:
  마지막 메시지 미리보기
  room unread count
  참여 전 메시지 내용
```

참여 전 메시지 preview를 보여주지 않는 이유는 현재 참여 구간 이후 메시지만 조회할 수 있다는 정책과 맞추기 위해서다.

### 5.3 주요 행동

```text
방 목록 탐색:
  사용자가 참여 가능한 room을 찾는다.

방 상세 확인:
  사용자가 참여 여부를 판단한다.

참여하기:
  참여를 요청한다.
  성공하면 참여 상태를 JOINED로 보정하고 채팅 페이지 진입을 허용한다.

채팅 페이지 진입:
  참여 성공 이후 가능하다.
```

이미 참여 중인 room이 조회되거나 선택되면 참여 가능한 방 흐름이 아니라 내가 참여한 방 흐름으로 보정할 수 있다.

### 5.4 실패와 보정

```text
참여 가능한 방 목록 조회 실패:
  기존 목록을 유지하거나 빈 결과와 구분되는 실패 상태를 표시한다.

방 상세 확인 실패:
  참여 버튼을 확정적으로 허용하지 않는다.

참여 요청 진행 중:
  중복 참여 요청을 막고 진행 중 상태를 보여준다.

참여 요청 실패:
  참여 전 상태로 되돌리고 실패 이유를 사용자가 이해할 수 있게 표현한다.

운영 개입 중인 room:
  신규 참여가 제한될 수 있으므로 서버 결과를 따른다.
```

## 6. 방 채팅 페이지

방 채팅 페이지는 사용자가 참여 중인 room에서 메시지를 조회하고 전송하는 페이지다.

이 페이지의 핵심 가치는 메시지 송수신, message read receipt count, 실시간 동기화 상태를 사용자가 이해할 수 있게 만드는 것이다.

### 6.1 진입 흐름

```text
사용자가 방 채팅 페이지에 진입
  -> room 상세 또는 참여 상태 확인
  -> 초기 메시지 구간 조회와 readerKey 기반 read cursor snapshot 조회를 병렬로 실행
  -> 초기 표시 위치 결정
  -> message read receipt count 계산
  -> 실시간 메시지 수신 준비
  -> 현재 화면을 보고 있다는 신호 전송
```

채팅 페이지는 현재 사용자가 해당 room에 참여 중인 경우에만 사용할 수 있다.

사용자가 room을 나간 뒤 재입장하면 이전 참여 구간 메시지를 현재 메시지로 이어 보지 않는다.

방 목록의 room unread count는 채팅방 진입 여부를 판단하는 요약 정보다.

채팅방에 진입한 뒤 어떤 메시지부터 보여줄지는 메시지 조회 결과를 기준으로 결정한다.

### 6.2 표시 정보

```text
표시 대상:
  메시지 목록
  메시지 전송 중 상태
  메시지 전송 실패 상태
  메시지 수정 상태
  메시지 삭제 상태
  message read receipt count
  실시간 연결 상태
  catch-up 필요 상태

표시하지 않는 대상:
  삭제된 메시지의 content
  삭제된 메시지 placeholder
  현재 참여 구간 밖의 메시지
```

삭제된 메시지는 content를 노출하지 않는다.

초기 UX에서는 "삭제된 메시지입니다" 같은 placeholder도 표시하지 않는다.

### 6.3 메시지 조회 흐름

```text
초기 조회:
  현재 참여 구간 기준으로 메시지를 조회한다.
  읽지 않은 메시지가 있으면 첫 번째 안 읽은 메시지를 기준으로 화면을 연다.
  읽지 않은 메시지가 없으면 최신 메시지 기준으로 화면을 연다.

사용자 탐색:
  현재 참여 구간 안에서 이전 메시지를 추가 조회한다.

누락 복구:
  재연결 또는 메시지 순서 누락 감지 후 누락 복구를 수행한다.
```

메시지는 서버가 확정한 room 안의 메시지 순서를 기준으로 정렬한다.

중복 수신된 메시지는 같은 메시지로 병합한다.

### 6.4 메시지 전송 흐름

```text
사용자가 메시지 입력
  -> 전송 가능 여부 확인
  -> 화면에 전송 중 메시지 표시
  -> 메시지 전송 요청
  -> 성공하면 전송 중 메시지를 전송 완료 메시지로 표시
```

현재 기기에서 보낸 메시지는 전송 요청의 성공 결과를 기준으로 전송 완료 상태가 된다.

다른 기기에서 보낸 내 메시지나 다른 참여자가 보낸 메시지는 실시간 이벤트로 현재 채팅 페이지에 추가된다.

이 경우 현재 기기에는 대응되는 전송 중 메시지가 없을 수 있으므로, 수신한 메시지는 기존 메시지와 중복되지 않게 병합하거나 새 메시지로 추가한다.

전송 실패 시 사용자가 재시도하거나 실패 메시지를 제거할 수 있어야 한다.

실시간 연결이 끊겨도 메시지 전송 가능 여부는 별도 정책으로 판단한다.

단, 실시간 동기화가 지연될 수 있음을 화면에서 표현할 수 있어야 한다.

### 6.5 메시지 수정과 삭제 흐름

```text
메시지 수정:
  수정 가능 여부 확인
  메시지 수정 요청
  성공 결과, 실시간 이벤트, 재조회 결과로 보정

메시지 삭제:
  삭제 가능 여부 확인
  메시지 삭제 요청
  성공 결과, 실시간 이벤트, 재조회 결과로 보정
```

수정과 삭제는 서버의 최종 권한 판단을 따른다.

클라이언트는 명백히 불가능한 수정/삭제 행동을 사전에 제한할 수 있다.

서버 확정 전 상태와 서버 확정 후 상태는 구분한다.

### 6.6 읽음 흐름

```text
채팅 페이지 진입:
  현재 채팅 페이지를 보고 있다는 신호를 보낸다.

사용자가 채팅 페이지를 보고 있음:
  보고 있는 상태를 유지한다.

채팅 페이지 이탈:
  더 이상 보고 있지 않다는 신호를 보낸다.

마지막으로 읽은 위치 동기화가 필요한 경우:
  클라이언트가 foreground 복귀, reconnect, scroll-to-bottom, 명시적 동기화 시점에
  현재 사용자가 어디까지 읽었는지 서버에 알려 동기화할 수 있다.

다른 사용자의 읽음 위치 변경 수신:
  화면에 보이는 메시지의 message read receipt count를 다시 계산할 수 있다.

message read receipt count snapshot 보정:
  클라이언트가 채팅방 진입, 재연결, foreground 복귀 시
  readerKey 기반 read cursor snapshot을 서버에서 다시 가져온다.
  snapshot을 교체한 뒤 현재 화면의 message read receipt count를 다시 계산한다.
```

클라이언트는 매 메시지마다 읽음 위치 갱신을 요청하지 않는다.

채팅 페이지를 보고 있는지에 대한 활성화/비활성화 신호가 기본 읽음 신호다.

읽음 위치 동기화 요청은 사용자가 읽은 마지막 지점을 서버에 명시적으로 알려야 하는 경우에 사용하는 보조 흐름이다.

### 6.7 실시간 연결과 복구 흐름

```text
채팅 페이지 진입:
  해당 room의 실시간 수신을 활성화한다.

실시간 이벤트 수신:
  메시지, room 상태, 읽음 상태를 빠르게 반영한다.

실시간 이벤트 중복:
  메시지 식별자, 메시지 순서, 수정 버전 기준으로 병합한다.

실시간 이벤트 누락 감지:
  누락 복구 필요 상태로 전환한다.

reconnect:
  실시간 수신을 복구하고 필요한 범위를 보정한다.
```

실시간 연결 실패는 곧바로 메시지 조회 실패를 의미하지 않는다.

하지만 실시간 동기화 지연 상태는 사용자가 이해할 수 있게 표현해야 한다.

### 6.8 방 나가기 흐름

```text
사용자가 방 나가기 선택
  -> leave 가능 여부 확인
  -> 방 나가기 요청
  -> 보고 있지 않다는 신호 전송
  -> 실시간 수신 해제
  -> 성공 시 채팅 페이지를 더 이상 사용 가능 상태로 두지 않음
  -> 내가 참여한 방 목록 페이지로 복귀
```

방 나가기 성공 후에는 해당 room의 현재 채팅 페이지에서 메시지 전송, 수정, 삭제, 추가 조회를 허용하지 않는다.

## 7. 페이지 전환 흐름

```text
내가 참여한 방 목록 페이지
  -> 방 선택
  -> 방 채팅 페이지

참여 가능한 방 조회 페이지
  -> 방 상세 확인
  -> 참여 성공
  -> 방 채팅 페이지

방 채팅 페이지
  -> 방 나가기 성공
  -> 내가 참여한 방 목록 페이지

방 채팅 페이지
  -> 뒤로가기 또는 목록 이동
  -> 내가 참여한 방 목록 페이지
```

페이지 전환 중에도 진행 중인 요청과 실시간 이벤트가 도착할 수 있다.

클라이언트는 현재 페이지에 직접 필요한 상태와 전역적으로 보정해야 하는 상태를 구분해야 한다.
